# 一、JDBC如何与mysql进行交互

JDBC与数据库连接基本分为以下几步

1、注册驱动（一次即可）

2、配置数据库连接信息进行连接

3、构建运行sql的语句（Statement）

4、运行语句

5、处理运行成果（ResultSet）

6、释放资源

我们首先进行环境的配置，使用Maven导入mysql依赖

使用Maven创建项目时会有pom.xml文件，将我们的mysql依赖写入即可

<dependencies>
    <dependency>
        <groupId>mysql</groupId>
        <artifactId>mysql-connector-java</artifactId>
        <version>8.0.33</version>
    </dependency>
</dependencies>

随后编写程序，为了方便记忆，我将一步步拆分步骤来演示如何创建连接

# 1、注册驱动(一次即可)

没什么说的，固定格式，即加载一个驱动。但在较新项目中可以省略不写，如spring boot中

Class.forName("com.mysql.cj.jdbc.Driver");

# 2、配置数据库连接信息进行连接

数据库基本信息，URL中?后的东西最好加上，为了防止乱码等问题。

String url = "jdbc:mysql://localhost:3306/pkxss?useUnicode=true&characterEncoding=utf-8";
String user = "root";
String password = "123456789";
Connection conn = DriverManager.getConnection(url,user,password);

# 3、构建运行sql的语句（Statement）

建立连接，写入sql语句

Statement stmt = conn.createStatement();
String sql = "select * from users where id = 1";

# 4、运行语句

ResultSet result = stmt.executeQuery(sql);

# 5、处理运行成果（ResultSet）

while (result.next()){
    System.out.println(result.getString("id"));
    System.out.println(result.getString("username"));
    System.out.println(result.getString("password"));
    System.out.println(result.getString("level"));
}

result.next()即将指针向下移动一行，如果有值返回true,否则返回false。
在此处即如果有值相当于while(true)一直循环知道false

result.getString获取某一列的值，注意这里的列一定要和列名对应，并且数据类型也要对应，不清楚的话用String准没错

# 6、释放资源

result.close();
stmt.close();
conn.close();

完整代码展示

package com.example;

import com.mysql.cj.jdbc.Driver;

import java.sql.*;

public class JDBCTest {
    public static void main(String[] args) throws Exception {
        Class.forName("com.mysql.cj.jdbc.Driver");
        String url = "jdbc:mysql://localhost:3306/pkxss";
        String user = "root";
        String password = "123456789";
        Connection conn = DriverManager.getConnection(url,user,password);
        Statement stmt = conn.createStatement();
        String sql = "select * from users where id = 1";
        ResultSet result = stmt.executeQuery(sql);
        while (result.next()){
            System.out.println(result.getString("id"));
            System.out.println(result.getString("username"));
            System.out.println(result.getString("password"));
            System.out.println(result.getString("level"));
        }
        result.close();
        stmt.close();
        conn.close();
    }
}

# 二、白盒视角下存在SQL注入常见点

我们讲完了如何用jdbc与数据库建立连接，接下来我们来说下常见的一些导致sql注入的写法

# （一）Statement与PreparedStatement

Statement与PreparedStatement是孪生的一对双胞胎，PreparedStatement使用了预编译，大大降低了sql注入的风险，Statement是Java JDBC下执行SQL语句的一种原生方式，执行语句时需要通过拼接来执行。若拼接的语句没有经过过滤，将出现SQL注入漏洞

什么是预编译呢？预编译有什么好处?
预编译顾名思义，它先将我的sql语句的骨架传输过去并缓存起来，当我们修改传参时，只需要将带入参数来查询语句即可
select * from users where id = 1
比如这个sql语句，如果没有使用预编译，每次进行查询时，服务器需要来判断sql语法是否正确？是否有users这个表？
然后再去执行查数据

但使用预编译后，数据库会将骨架传过去，我们只要修改参数即可
String sql = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setInt(1, userId);
pstmt.executeQuery();

这样有两个好处：
速度快（性能高）
尤其是在批量操作（比如循环插入 1000 条数据）时，只编译一次，执行 1000 次，比编译 1000 次要快得多。
安全（防SQL注入）
因为 SQL 的结构（骨架）在预编译时就已经定死了，后面传进去的参数永远只是“数据”，不可能被数据库当成 SQL 代码来执行。

我们来看一下问题所在

String id = "1";//(该参可控用户可控)
String sql = "select * from users where id = " + id;
Statement stmt = conn.createStatement();
ResultSet result = stmt.executeQuery(sql);

该示例的sql语句是让用户使用id查询用户信息。但这里使用的是拼接参数方式，并且使用的statement，因为statement允许用户拼接参数，或者说只能这样拼接参数，所以导致sql注入的风险

那我们来看一下安全的写法是怎么样的

String sql = "select * from users where id = ?";
String id = "2 ' or1=1" ;
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, id);
ResultSet result = pstmt.executeQuery();

我们使用prepareStatement进行预编译，?当作占位符，使用setString来传入参数，此时传入的参数只会当作“数据”，不会被当作sql语句执行

我们来查看一下语句

"C:\Program Files\Java\jdk-11.0.11\bin\java.exe" "-javaagent:F:\JetBrains\IntelliJ IDEA Community Edition 2024.1.4\lib\idea_rt.jar=8676:F:\JetBrains\IntelliJ IDEA Community Edition 2024.1.4\bin" -Dfile.encoding=UTF-8 -classpath F:\JetBrains\untitled\target\classes;C:\Users\syz\.m2\repository\com\mysql\mysql-connector-j\8.0.33\mysql-connector-j-8.0.33.jar;C:\Users\syz\.m2\repository\com\google\protobuf\protobuf-java\3.21.9\protobuf-java-3.21.9.jar;C:\Users\syz\.m2\repository\p6spy\p6spy\3.9.1\p6spy-3.9.1.jar com.example.JDBCTest

看起来像是sql注入成功了，但其实这是假的。数据库只会将String id = "2' or '1'='1 " ;中的参数当作一个字符串，实际上的sql语句：select * from users where id = "'2' or '1'='1'"

或者我们用预编译来理解，他会将所有的'预编译为'

即select * from users where id = '1' or '1'='1' 其与 select * from users where id = '1 or 1=1' 基本一样(底层肯定不一样，不过在此场景我们用来理解将其认为是相同的)

那是不是使用prepareStatement就没有问题了？prepareStatement本身是没有问题的，但是有些拼接错误，导致依旧存在sql注入

# （二）prepareStatement拼接问题

那是不是使用prepareStatement就没有问题了？prepareStatement本身是没有问题的，但是有些拼接错误，导致依旧存在sql注入

String id = "1" ;
String sql = "select * from users where id = '" + id + "'";
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();

我们可以看下这个代码，同样是用prepareStatement来进行预编译执行sql语句，但是由于没有使用占位符，使用动态字符的拼接，导致依旧会存在sql注入漏洞

String id = "1' or 1=1 -- -" ;
String sql = "select * from users where id = '" + id + "'";
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();

com.mysql.cj.jdbc.ClientPreparedStatement: select * from users where id = '1' or 1=1 -- -'
admin
user1
user2

原因：此处的问题在于拼接并且绕过了预编译，预编译无法识别sql注入，它只能将你的代码看作骨架传入，并对传入的参数进行预编译。而这样写，在prepareStatement之前就依旧注入进去了，prepareStatement会将

select * from users where id = '1' or 1=1 -- -';

作为骨架传入，由于没有 ? 它根本不会进行数据的预编译，直接执行sql语句导致注入。所以最安全的方法即为使用 ？占位符+prepareStatement预编译。

# （三）order by / group by

order by / group by会造成sql注入的原因在于他们无法使用预编译占位符。

我们可以来看一下这个例子

String sql = "SELECT * FROM users";
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();

admin
3
user1
1
user2
2

String sql = "SELECT * FROM users ORDER BY level";
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();

user1
1
user2
2
admin
3

String sort = "level";
String sql = "SELECT * FROM users order by " + sort;
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();

user1
1
user2
2
admin
3

我们正常输出，不排序的情况下，admin在第一位；当我们使用level来进行排序时，由于admin的level为3，所以他排在最后。那我们来使用占位符尝试一下。

String sql = "SELECT * FROM users order by ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
String sort = "level";
pstmt.setString(1, sort);
ResultSet result = pstmt.executeQuery();

admin
3
user1
1
user2
2

我们惊奇的发现，最后的结果没有被排序。这是因为level被当作"数据"去查询了;但在这个排序语句里，order by level还是sql语句，所以他执行成了如下语句，所以不会排序。

SELECT * FROM users order by 'level'

所以，order by /group by这种字段的防护非常单一，就是白名单校验。不符合白名单的一律拦截。反之，如果order by 参数可控，并且没有白名单，那么他99%会有问题

# （四）like模糊查询

like的模糊查询同样也无法正常使用预编译，我们来看下示例

String sort = "a";
String sql = "SELECT * FROM users where username like '%" + sort + "%' ";
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();


com.mysql.cj.jdbc.ClientPreparedStatement: SELECT * FROM users where username like '%a%' 
admin
3

我们来使用字符串的直接拼接来尝试模糊查询，发现可以正常查询。但我们知道，这种拼接绝对会造成SQL注入的风险，当我们输入恶意payload即会造成注入

String sort = "' or 1=1 #";
String sql = "SELECT * FROM users where username like '%" + sort + "%' ";
PreparedStatement pstmt = conn.prepareStatement(sql);
ResultSet result = pstmt.executeQuery();

com.mysql.cj.jdbc.ClientPreparedStatement: SELECT * FROM users where username like '%' or 1=1 -- -%'

1
user2

那我们尝试下预编译，使用占位符会怎么样呢？

String sort = "a";
String sql = "SELECT * FROM users where username like '%" + "?" + "%'";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1,sort);
ResultSet result = pstmt.executeQuery();

所以，like无法直接使用占位符来进行预编译。

那么，like该如何安全的来进行查询的-----即通过预拼接

String sort = "a";
String keyword = "%" + sort + "%";
String sql = "SELECT * FROM users where username like ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1,keyword);
ResultSet result = pstmt.executeQuery();

com.mysql.cj.jdbc.ClientPreparedStatement: SELECT * FROM users where username like '%a%'
admin
3

此时用户可以控制sort，但是写入恶意payload也无妨，因为预编译会将keyword当作"数据"来处理，不会当成SQL的结构来执行
